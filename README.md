# Bauko Land Valuation Map

Next.js + react-leaflet app that visualises the **Bauko (Mountain Province)
2027 Schedule of Base Unit Market Values for Residential, Commercial, and
Industrial Lands** on top of OpenStreetMap. Built to work fully offline once
tiles are pre-downloaded.

> Looking for context on **other tools in this space** (MapX, Infoman
> RISE, BLGF's RA 12001 mandate, what's unique about this project)?
> See [RELATED_WORK.md](./RELATED_WORK.md).

## Quick start (online — for collaborators)

```bash
git clone <repo-url>
cd leaflet-test-app
npm install
npm run dev
```

Open <http://localhost:3000>. The app opens in **Online** mode by default and
streams basemap tiles directly from OpenStreetMap, so a fresh clone works
without any extra setup. Switch municipalities via the dropdown next to
**BAUKO** in the top-left — direct links work too (`/?m=barlig`,
`/?m=tadian`, `/?m=sagada`).

**Note:** `public/tiles/` is intentionally **not** in the repo (it's the
offline OSM tile pack, hundreds of MB, easy to regenerate). The app falls
back to online OSM whenever those tiles are absent.

### Edit-mode flows

- Toggle edit mode with the pencil icon in the top-right nav
- The editor panel is **draggable** (drag the header bar) and
  **collapsible** (▾ button) — layout preference is per-municipality and
  persists in localStorage
- Click a road segment on the map to select it; click a class chip to
  bake the selection into a corridor polygon
- Selecting a polygon enables vertex editing on **only** that polygon
  (no more global-edit perf hit)
- The Cut tool only operates on the currently-selected polygon

## Quick start (offline-first, optional)

If you want the basemap to work without internet (field consultations,
LGU office Wi-Fi, etc.), download a tile pack once:

```bash
cd leaflet-test-app
npm install
npm run offline:prepare   # one-time: boundaries + tiles + production build
npm run dev
```

The app then defaults to online tiles, but you can switch to offline via
the settings (gear) icon in the top-right nav once tiles are available.
If you only want a fast coarse test, run `npm run offline:prepare:quick`
instead; it downloads zooms 10-14 only.

## What the map shows

- **Bauko municipal outline** — derived from the union of the barangay
  polygons so the outer boundary and barangay edges match.
- **Bauko's 22 barangays** — outlined (no fill) so the basemap stays
  readable. Click any barangay for a popup with its name, default
  inner-lots classification, and per-square-metre value from the 2027
  schedule.
- **Mountain Province municipality tabs** — Bauko is the detailed working tab;
  the other municipalities are lightweight context tabs that render only the
  selected municipal outline.
- **Custom zones (drawn manually)** — see "Drawing zones" below.

The Schedule's road-corridor tiers (C-1, C-2, R-1, R-2) follow road geometry
rather than barangay shape, so they are **not** auto-coloured. Use the
drawing toolbar to trace those zones along provincial / national / all-weather
roads and tag each one with its sub-classification.

## Custom SMV zones

`public/data/bauko_zones.geojson` is a FeatureCollection of authoritative SMV
polygons. Each feature must have `properties.classification` set to one of
`C-1`, `C-2`, `C-3`, `R-1`, `R-2`, `R-3`, `R-4`, or `R-5`.

The app reads this file when `public/data/sources.json` has `"has_zones": true`
and renders each feature in the colour of its class. Polygons can be:

- drawn in QGIS and pasted in,
- drawn via the in-app pencil tool and exported with **Export GeoJSON**, or
- generated by hand directly in this file.

The zones layer is separate from `bauko_barangays_custom.geojson`, which is the
barangay administrative boundary override. Zones overlay the barangays; they do
not replace barangay boundaries.

## Drawing zones

There's a panel at the bottom-left:

1. Pick a class (R-1 … R-5 or C-1 … C-3) — that becomes the active tag.
2. Set **Corridor width** if you are tracing a road corridor. The default is
   30 m on each side of the line.
3. Use the **leaflet-geoman toolbar** at the top-left. The polyline tool traces
   a road and converts it to a buffered polygon; the polygon / rectangle tools
   draw a freeform zone directly.
4. The shape is saved automatically to `localStorage` as you draw and edit.
5. **Export GeoJSON** writes a `bauko_zones.geojson` file you can keep in the
   repo or share. **Import** loads one back. **Clear** removes all custom
   zones.

To make edits permanent/offline-bundled, export `bauko_zones.geojson` and put
it at `public/data/bauko_zones.geojson`. The editor loads that file on first
open. Once you start editing in the browser, your local browser copy takes
priority so work is not lost between reloads.

## Saved viewports (barangay + per-stretch)

You can capture a custom map framing for any barangay or, when the
schedule has sub-stretch detail, for any individual stretch under a
barangay. Clicking the same row later flies the map back to that exact
spot.

**Hierarchy of viewports the app uses**, in order of precedence:

1. **Active stretch's saved view** — landmark-level, the most specific.
2. **Active barangay's saved view** — falls back to this when the
   stretch hasn't been framed yet.
3. **Auto-fit to the barangay polygon** — Bauko default when neither
   of the above is set.

**How to save a stretch viewport** (works on `/?m=bontoc` today, on
any municipality whose schedule has `stretches` data):

1. In the sidebar, click a class → its barangay → one of the sub-items
   that appears (e.g. *"National Road: Circle → Kalangeg building"*
   under Poblacion → C-1).
2. The row highlights blue. Pan / zoom the map to frame that exact
   stretch the way you want it shown.
3. Click **"Save view"** in the top nav. The current map center,
   zoom, and bounding box are captured.
4. A 📍 icon appears next to the row in the sidebar — that's the
   visual hint that the stretch has a saved viewport.
5. Click any other row, then come back: the map flies back to the
   saved framing automatically.

**How the system knows which stretch a saved view belongs to**

When you click a stretch row, the app composes a stable identifier
of the form `${classId}|${barangaySlug}|${stretchIdx}` (e.g.
`c-1|poblacion|0` for the first stretch under Poblacion C-1). That
key is used to:

- Track which stretch is the currently active selection.
- Look up the saved viewport in `localStorage` under
  `smv-stretch-view-v1:<municipality-slug>`.
- Mark the row with a 📍 if a saved view exists.

Saving a view writes `savedStretchViews[key] = { center, zoom, bbox }`
to `localStorage`. Loading a stretch reads the same key.

**Top-nav save / reset buttons are context-aware**

- A stretch is selected → Save / Reset operates on the **stretch**
  view.
- No stretch selected but a barangay is → Save / Reset falls back to
  the **barangay** view (original behaviour).

So one set of buttons handles both levels — you don't think about
which scope is in play; whatever's selected most specifically gets
the save.

**Caveats**

- Viewports are persisted **per browser** via `localStorage`. They
  don't sync between devices or to coworkers. If we ever want
  shareable viewports, the natural path is to store them as feature
  properties inside the GeoJSON so they travel with the data.
- Switching class, group, or barangay clears `activeStretchIdx`, so
  the buttons revert to the barangay-level scope automatically. No
  way to accidentally save the wrong view.

## Going offline

Boundaries and valuation data are bundled in `public/data/`, so they work
offline after install. The basemap needs one extra preparation step: download
tiles into `public/tiles/` while you still have internet.

The offline basemap is still OpenStreetMap. `offline:prepare` downloads OSM
raster tiles for the Bauko area and stores them locally, so roads, place names,
rivers, terrain styling, and other labels baked into those tiles remain visible
without internet.

### One-command setup

```bash
# Full local Bauko pack: boundaries + z10-16 OSM tiles + build
npm run offline:prepare

# Faster coarse pack for testing: boundaries + z10-14 tiles + build
npm run offline:prepare:quick
```

After that, run the app normally:

```bash
npm run dev
# or, after npm run build:
npm run start
```

Open <http://localhost:3000> and leave the top-right tile toggle on
**Offline**. No outbound connection is needed once the tiles exist locally.

### One-time tile download

```bash
# Bauko bbox at z10–16 (default — 1,601 tiles, roughly 30 minutes)
npm run tiles:bauko

# Bauko at z10–17 (denser detail, much longer)
npm run tiles:bauko:hires

# All of Mountain Province at z10–14 (overview only)
npm run tiles:mp

# List supported bbox keys, including individual municipalities
npm run tiles:list

# Download one municipality only, for example Sagada at z10–14
node scripts/download-tiles.mjs --bbox sagada --zooms 10-14
```

Tiles are written to `public/tiles/{z}/{x}/{y}.png`, served at
`/tiles/...` by Next.js. The script:

- respects a 1.1-second delay per request (configurable with `--delay`),
- sends a clear `User-Agent` and skips files it already downloaded, and
- only covers the bbox you specify.

The tile script auto-loads bbox keys for every Mountain Province municipality
from `public/data/mountain_province_municipalities.geojson`, adding `0.015°`
padding around each municipality. To list available keys:

```bash
node scripts/download-tiles.mjs --list-bboxes
```

Then download any municipality by key:

```bash
node scripts/download-tiles.mjs --bbox sagada
node scripts/download-tiles.mjs --bbox bontoc
```

Rerunning a tile command does **not** redownload everything. Existing tile
files are skipped, so editing SMVs or patching app code will not touch the
offline basemap cache.

Recommended offline strategy:

- Download Bauko at detail with `npm run tiles:bauko`.
- The app lets you zoom beyond the downloaded native tile level; if only z16
  tiles are available, zooms 17-18 reuse z16 tiles and look softer.
- For sharper deep zoom, run `npm run tiles:bauko:hires` to download z17.
- Download Mountain Province overview with `npm run tiles:mp`.
- Add detailed tiles for a specific municipality only when needed, e.g.
  `node scripts/download-tiles.mjs --bbox bontoc --zooms 10-16`.
- Avoid downloading all Mountain Province at z16/z17 unless you truly need it;
  that becomes many thousands of tiles.

Mapbox offline hybrid (higher clarity):

- `npm run tiles:bauko:mapbox:hires` downloads Bauko-only Mapbox hybrid tiles at
  z10–18 with hi-DPI (`@2x`) enabled.
- `npm run tiles:mp:mapbox:hires` does the same for all Mountain Province
  (much heavier download).
- These scripts write to `public/tiles-mapbox-hidpi/` (served at
  `/tiles-mapbox-hidpi/...`) so old non-retina cache files in `public/tiles-mapbox/`
  won't be reused accidentally.
- If offline Mapbox looks soft, the usual cause is older 256px cached tiles.
  Re-run one of the hi-res commands above and refresh the page.
- For a lighter consultation pack while preserving building-level detail, use
  `npm run tiles:bauko:mapbox:smart` (full z10–16, smart-clipped z17–18 around
  zones/frontage/roads).

Bauko offline vector map (lightweight):

- The settings menu includes **Offline Vector (Bauko)**. It reads
  `public/data/bauko.pmtiles` and renders a crisp vector basemap in Leaflet.
- Download a Bauko-only PMTiles extract by setting `PROTOMAPS_SOURCE` to the
  current Protomaps planet build URL, then running:
  `PROTOMAPS_SOURCE=https://build.protomaps.com/<build>.pmtiles npm run tiles:bauko:vector`
- This is much smaller than raster satellite. It shows roads, labels, rivers,
  landuse, and OSM building footprints where available. It is not satellite
  imagery, so it will not show roof photos.

All municipalities (batch mode):

- OSM detailed (z10–16) for all Mountain Province municipalities:
  `npm run tiles:all`
- OSM quick overview (z10–14) for all municipalities:
  `npm run tiles:all:quick`
- Mapbox hybrid hi-res (z10–18, very heavy) for all municipalities:
  `MAPBOX_TOKEN=... npm run tiles:all:mapbox:hires`
- Mapbox hybrid smart (z10–18, clipped at z17+ to zones/frontage/roads):
  `MAPBOX_TOKEN=... npm run tiles:all:mapbox:smart`

Batch mode runs municipalities one-by-one and reuses existing cached tile
files, so interrupted runs can be resumed safely by re-running the same
command.

> **Note**: OSM's
> [Tile Usage Policy](https://operations.osmfoundation.org/policies/tiles/)
> strongly discourages bulk downloads. Pulling a small region for a one-off
> offline workflow is generally fine; doing this regularly or for a large
> region is not. For heavy use, host your own tile server or use a paid
> provider (Stadia, MapTiler, Thunderforest, etc.) — swap the URL in
> `components/LeafletMap.js`.

### Running fully offline

After downloading tiles, keep the **Offline** toggle selected. The app serves
the basemap from `/tiles/{z}/{x}/{y}.png` and boundaries from
`/data/*.geojson`. The edit/drawing tools also work offline; drawn zones are
saved in the browser and can be exported as GeoJSON.

## Boundary source

The bundled barangay boundaries are fetched from the public GeoRiskPH ArcGIS
service backed by PSA barangay boundary data:

```bash
npm run boundaries:fetch
```

That command writes:

- `public/data/bauko_barangays.geojson` — Bauko's 22 barangays
- `public/data/bauko.geojson` — municipality outline derived from the union of
  those barangays

The app always loads those bundled files on page load. The upload control can
preview another GeoJSON temporarily, but refresh/reset returns to the bundled
PSA/GeoRiskPH files.

`public/data/sources.json` has `has_custom_barangays`:

- `true` loads `bauko_barangays_custom.geojson`, for QGIS edits merged via
  `npm run barangays:merge`.
- `false` loads the bundled PSA 2019 defaults.

Flip `has_custom_barangays` manually depending on whether custom barangay edits
are ready to display. The current public-consultation build leaves it `false`.

## Replacing the bundled barangay boundaries

If you get official LGU boundary data later, replace the bundled files in one
of these ways:

### Option A — Upload via the UI (easiest)

If your file is already GeoJSON (`.geojson`), or you can convert it on
[mapshaper.org](https://mapshaper.org/) (drag in your `.shp` + `.dbf`,
choose Export → GeoJSON):

1. Click **Upload .geojson…** in the top-left "Boundaries" panel.
2. Pick your file.
3. The map swaps to your boundaries for the current session.
   Click **Reset to default** or refresh the page to revert.

### Option B — Regenerate from the public PSA/GeoRiskPH source

If you want to refresh the bundled coordinates from the current GeoRiskPH
service:

```bash
npm run boundaries:fetch
```

This overwrites `public/data/bauko_barangays.geojson` and `public/data/bauko.geojson`.

### Option C — Per-barangay edits in QGIS

1. In QGIS, select the barangay layer and Export → Save Features As… → GeoJSON, CRS = EPSG:4326. Per-feature splits like `NAME_3_<name>.geojson` (from "Split Vector Layer by Attribute") work too — drop them anywhere under `data-raw/`.
2. Run `npm run barangays:merge`. The script:
   - walks `data-raw/` (recursively) for `*.geojson`,
   - matches each feature to a barangay by name, replacing the PSA default,
   - **clips every barangay polygon to `public/data/bauko.geojson`** so external edges line up with the Bauko outline,
   - writes `public/data/bauko_barangays_custom.geojson`.
3. If you want those edits to become the default runtime layer, replace
   `public/data/bauko_barangays.geojson` with the merged output.

The clip step uses `@turf/turf`, so make sure `npm install` has run.

To replace the **municipal outline** directly, overwrite
`public/data/bauko.geojson`. Prefer deriving it from the barangay union so
the outer boundary and barangay edges stay aligned.

### Project layout for source data

- `data-raw/` — raw inputs (shapefiles, GPKG, KML, etc.) that are not loaded
  by the app directly. Keep originals here so they can be re-converted.
- `public/data/` — the GeoJSON files the app actually reads. Custom files
  named `*_custom.geojson` override the bundled defaults.

Currently set up overrides:

| File | Purpose |
|---|---|
| `bauko.geojson` | Bauko municipal outline derived from the barangay union |
| `bauko_barangays.geojson` | the 22 runtime barangay polygons |

### Option D — Convert a shapefile in place

If you have raw `.shp` + `.dbf` (and ideally `.prj` set to WGS84):

```bash
npm run import:shapefile -- path/to/bauko_brgys.shp
# optional: --name-prop Brgy_Name --out public/data/bauko_barangays_custom.geojson
```

The converter auto-detects common name property keys (`NAME`, `Brgy_Name`,
`ADM4_EN`, etc.); pass `--name-prop` if yours uses something different.
The output goes to `public/data/bauko_barangays_custom.geojson` by default.

> If your shapefile is **not** in WGS84, reproject it first (e.g. with
> `ogr2ogr -t_srs EPSG:4326 out.shp in.shp`). Leaflet expects lon/lat.

## Data sources

- **Municipal & barangay polygons** — fetched with
  `scripts/fetch-psa-boundaries.mjs` from the public GeoRiskPH ArcGIS REST
  service's PSA Barangay layer, filtered to Bauko's 22 barangays. The
  municipal outline is derived from the union of those barangays.
- **Province polygon** — same source, CAR provinces file.
- **Land valuation data** — `public/data/bauko_valuations.json`, transcribed
  from the 2027 Schedule supplied for this project.

## Notes on the inner-lots classification

The default colouring assumes each barangay's *inner lots* (i.e. the bulk
of its area, away from main roads) per the 2027 schedule:

| Class | Barangays |
|-------|-----------|
| R-3 (₱4,030/m²) | Mabaay, Sinto, Sadsadan, Poblacion, Guinzadan Central, Guinzadan Sur, Mount Data, Monamon Sur |
| R-4 (₱3,150/m²) | Monamon Norte, Banao, Otucan Sur, Otucan Norte, Tapapan, Leseb, Lagawa |
| R-5 (₱2,480/m²) | Balintaugan |
| Unclassified | Abatan (mixed/null per schedule), Bila, Guinzadan Norte, Mayag, Bagnen Proper, Bagnen Oriente |

Edit `lib/classifications.js` to change the map.
